## 1. 背景与动机

### 1.1 CP 并行概述

Context Parallelism (CP) 是处理长序列推理的关键并行策略。当单张卡无法容纳一个请求的全部 KV Cache 时，CP 将序列切分到多个设备上，每张卡只存储部分 KV，注意力计算时通过 all-gather 通信获取完整 KV 再进行计算。

vllm-ascend 在 `vllm_ascend/attention/context_parallel/` 下实现了一套完整的 CP 方案，覆盖：
- **PCP** (Prefill Context Parallelism)：长序列 prefill 阶段的上下文并行
- **DCP** (Decode Context Parallelism)：decode 阶段 KV cache 跨 rank 分布，query 数量少但 KV 分布在多卡

当前 `ascend_msmodeling` 已具备 TP/DP/EP/PP 四种并行的仿真能力（`tensor_cast/parallel_group.py`），但**缺少 CP 并行的仿真支持**。本 RFC 设计如何将 CP 并行引入仿真框架，并支持在不同推理引擎的 CP 实现间切换。

### 1.2 参考实现：vllm-ascend CP 架构

vllm-ascend 的 CP 实现核心架构如下：

```
                     ┌──────────────┐
     Long Sequence   │  PCP Manager │  Q 按 head/tail 切分
     Input Tokens ──►│  (pcp_utils) │──► 按 rank 分发
                     └──────────────┘
                            │
              ┌─────────────┼─────────────┐
              ▼             ▼             ▼
         PCP Rank 0    PCP Rank 1    PCP Rank N
              │             │             │
    ┌─────────┼─────────────┼─────────────┼────────┐
    │  all_gather(KV_cache) 跨 CP 组               │
    │  Q_head 做 attention (无 mask)               │
    │  Q_tail 做 attention (无 mask)               │
    │  all_gather(attn_output + softmax_lse)       │
    │  attention_update：merge head & tail outputs │
    └──────────────────────────────────────────────┘
```

**关键通信原语** (位于 `common_cp.py`)：
| 函数 | 作用 |
|------|------|
| `get_pcp_group().all_gather()` | 跨 PCP rank 聚合 KV cache |
| `get_dcp_group().all_gather()` | 跨 DCP rank 聚合 KV cache |
| `dist.all_to_all_single()` | DCP decode 阶段重排 query heads |
| `torch_npu.npu_attention_update()` | 合并多个 attention 的输出（基于 softmax LSE） |

**核心数据流**：
```
Step 1: all_gather KV → 每个 rank 获得完整 KV cache
Step 2: Q 按 head_idx/tail_idx 切分 → head/tail attention
Step 3: all_gather(attn_out + softmax_lse) → 聚合结果
Step 4: attention_update → 加权合并 head/tail 输出
```

**Attention 类型适配**：
- `sfa_cp.py` — Sparse Flash Attention (SFA) 的 CP 适配
- `mla_cp.py` — Multi-head Latent Attention (MLA，如 DeepSeek-V2/V3) 的 CP 适配
- `attention_cp.py` — 标准 Flash Attention 的 CP 适配

## 2. 当前模块冲突分析

`tensor_cast/parallel_group.py` 中的 `ParallelGroupType` 枚举：
```python
class ParallelGroupType(StrEnum):
    TENSOR_PARALLEL = "tensor_parallel"
    DATA_PARALLEL = "data_parallel"
    EXPERT_PARALLEL = "expert_parallel"
    PIPELINE_PARALLEL = "pipeline_parallel"
    # 缺少: CONTEXT_PARALLEL
```

`initialize_model_parallel` 中的 rank 布局假设：
```python
all_ranks.reshape(-1, dp_size, pp_size, ep_size, tp_size)
# 需要扩展为:
all_ranks.reshape(-1, dp_size, pp_size, cp_size, ep_size, tp_size)
```

**冲突点**：
1. `ParallelGroupType` 缺少 `CONTEXT_PARALLEL` 枚举
2. rank 网格布局不包含 CP 维度
3. `ParallelGroupManager` 无 `cp_group` 成员
4. Attn 模块仿真中无 CP 阶段的通信开销建模

## 3. Work Unit 分解

### Unit 1: 扩展 ParallelGroup 支持 CP

| 字段 | 内容 |
|------|------|
| **id** | `unit-001-cp-parallel-group` |
| **depends_on** | 无 |
| **scope** | `tensor_cast/parallel_group.py` |
| **acceptance_tests** | `cp_group.world_size` 正确反映 cp_size；rank 定位到正确的 cp group |
| **risk_level** | Tier 1 |
| **rollback_plan** | 还原 `ParallelGroupType` 和 `ParallelGroupManager.__init__` |

**实施要点**：
1. 在 `ParallelGroupType` 添加 `CONTEXT_PARALLEL = "context_parallel"`
2. 在 `initialize_model_parallel` 中将 rank 布局改为：`all_ranks.reshape(-1, dp_size, pp_size, cp_size, ep_size, tp_size)`
3. 添加 `self.cp_group` 及对应的 `o_proj_cp_group` / `mlp_cp_group` 等

### Unit 2: CP 通信建模仿真

| 字段 | 内容 |
|------|------|
| **id** | `unit-002-cp-comm-modeling` |
| **depends_on** | `unit-001` |
| **scope** | 新增 `tensor_cast/cp_sim.py` |
| **acceptance_tests** | all_gather 开销随 cp_size 和 seq_len 线性增长；attention_update 开销恒定 |
| **risk_level** | Tier 2 |
| **rollback_plan** | 删除 `cp_sim.py`，`ParallelGroupManager` 中的 cp_group 设为 None |

**实施要点**：
1. CP 阶段通信建模核心函数：

```python
def simulate_cp_attention_bw(
    seq_len: int,          # 序列长度 (本地切分后)
    head_dim: int,         # 注意力头维度
    num_heads: int,        # 注意力头数
    cp_size: int,          # CP 并行度
    kv_cache_dtype: torch.dtype,
    interconnect: InterconnectTopology,
) -> float:
    """返回 CP attention 步骤的总通信时间(秒)"""
    # Step 1: all_gather KV cache
    kv_bytes = seq_len * head_dim * num_heads * kv_cache_dtype.itemsize
    allgather_kv_time = allgather_comm_time(kv_bytes, cp_size, interconnect)
    
    # Step 2: attention compute (local)
    attn_compute_time = flash_attn_time(seq_len, head_dim, num_heads)
    
    # Step 3: all_gather attn output + LSE
    out_bytes = seq_len * head_dim * num_heads * 4  # float32
    allgather_out_time = allgather_comm_time(out_bytes, cp_size, interconnect)
    
    return allgather_kv_time + attn_compute_time + allgather_out_time
```

2. 支持两种 attention 类型：`"sfa"`, `"mla"`
3. 不同 attention 类型的 CP 通信模式差异：
   - **SFA CP**: Q 按 head/tail 切分各做一次 attention → 合并
   - **MLA CP**: 额外包含 `kv_b_proj` 操作 + `kv_tail_proj` 索引选取

### Unit 3: CP 仿真配置系统

| 字段 | 内容 |
|------|------|
| **id** | `unit-003-cp-config-system` |
| **depends_on** | `unit-001`, `unit-002` |
| **scope** | `tensor_cast/model_config.py`, `serving_cast/config.py` |
| **acceptance_tests** | 配置切换后 CP 参数正确传递；非法配置回退到默认行为 |
| **risk_level** | Tier 2 |
| **rollback_plan** | 移除 `CPConfig` dataclass，`ParallelConfig` 中 cp_size 默认值恢复 |

**配置类设计**：

```python
@dataclass
class CPConfig:
    """Context Parallelism 仿真配置"""
    cp_size: int = 1                          # CP 并行度
    cp_engine: str = "vllm_ascend"            # 推理引擎: "vllm_ascend" | "sglang" | "custom"
    attn_type: str = "flash_attn"             # attention 类型: "flash_attn" | "sfa" | "mla"
    cp_kv_cache_interleave_size: int = 128    # KV cache 交错块大小
    enable_pcp: bool = True                   # 启用 Prefill CP
    enable_dcp: bool = True                   # 启用 Decode CP
    cp_comm_ring: bool = True                 # Ring-based all-gather vs tree-based
    
    # 引擎特定参数
    engine_params: dict = field(default_factory=dict)
```

**引擎切换逻辑**：

```python
class CPEngine:
    """CP 引擎抽象基类 — 不同推理引擎的 CP 实现差异在此封装"""
    
    def cp_attention_flow(self, ...) -> list[CommOp]:
        """返回 CP attention 的通信算子序列"""
        raise NotImplementedError

class VllmAscendCPEngine(CPEngine):
    """vllm-ascend 风格：all_gather KV → Q split → attn → all_gather output → merge"""
    pass

class SglangCPEngine(CPEngine):
    """sglang 风格：ring-attention，环状传递 KV chunk"""
    pass
```

### Unit 4: 集成到 serving_cast 流水线

## 5. 配置示例

### 5.1 YAML 配置

```yaml
# ascend_msmodeling/configs/cp_vllm_ascend.yaml
parallel_config:
  world_size: 8
  tp_size: 2
  dp_size: 2
  cp_size: 2          # NEW: CP size = 2
  ep_size: 1
  pp_size: 1

cp_config:
  cp_engine: "vllm_ascend"
  attn_type: "flash_attn"
  cp_kv_cache_interleave_size: 128
  enable_pcp: true
  enable_dcp: true
```

### 5.2 引擎切换示例

```yaml
# 切换到 sglang ring-attention 风格
cp_config:
  cp_engine: "sglang"
  attn_type: "flash_attn"
  cp_kv_cache_interleave_size: 256
  enable_pcp: true
  enable_dcp: false    # sglang 可能仅支持 prefill CP
  engine_params:
    ring_degree: 4
    chunk_size: 8192
```

### 5.3 MLA 模型 CP 配置

```yaml
# DeepSeek-V3 的 MLA CP 仿真
cp_config:
  cp_engine: "vllm_ascend"
  attn_type: "mla"
  engine_params:
    kv_lora_rank: 512
    qk_rope_head_dim: 64
    v_head_dim: 128
```

## 6. 复杂度与风险总结

| 等级 | Work Unit | 关键风险 |
|------|-----------|----------|
| Tier 1 | Unit 1 (ParallelGroup) | rank 网格 reshape 可能影响现有并行组合 |
| Tier 2 | Unit 2 (Comm Modeling) | all_gather 开销估计精度依赖 interconnect 模型 |
| Tier 2 | Unit 3 (Config System) | 多引擎支持增加维护成本 |
| Tier 3 | Unit 4 (Integration) | CP + TP + DP 组合爆炸，需仔细验证边界 |
| Tier 2 | Unit 5 (Validation) | 真实硬件 CP 数据获取困难 |

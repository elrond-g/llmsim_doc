## 1. 背景与动机

### 1.1 CP 并行概述

Context Parallelism (CP) 是处理长序列推理的关键并行策略。当单张卡无法容纳一个请求的全部 KV Cache 时，CP 将序列按 token 维度切分到多个设备上，每张卡只存储部分 KV，注意力计算时通过集合通信获取完整 KV 再计算。

vllm-ascend 在 `vllm_ascend/attention/context_parallel/` 下实现了完整的 CP 方案，分为两层：
- **PCP**（Prefill Context Parallelism）：prefill 阶段，长序列的 Q 按 head/tail 切分后各自做 attention，最后通过 LSE 加权合并
- **DCP**（Decode Context Parallelism）：decode 阶段，KV cache 分布在多卡，每个 rank 只计算自己负责的那部分 KV 对应的 attention
- PCP 和 DCP 使用**独立的通信组**：`get_pcp_group()` 和 `get_dcp_group()`

当前 `ascend_msmodeling` 已具备 TP/DP/EP/PP 及 Ulysses 序列并行的仿真能力（`tensor_cast/model_config.py` `ParallelConfig`，`tensor_cast/parallel_group.py`），但**缺少 CP 并行的仿真支持**。本 RFC 设计如何将 CP 引入仿真框架，并支持在不同推理引擎的 CP 实现间切换。

### 1.2 参考实现：vllm-ascend CP 架构

#### 1.2.1 Rank 布局

vllm-ascend 的分布式 rank 布局（`parallel_state.py`）：

```python
# 实际 rank 网格: [ExternalDP, DP, PP, PCP, TP]
all_ranks = torch.arange(world_size).reshape(
    -1, global_dp_size, global_pp_size, global_pcp_size, global_tp_size
)
```

**注意**：vllm-ascend 的 rank 网格**没有独立的 EP 维度**；当 `transpose_tp_ep_enabled=True` 时，EP 等价分组通过 `torch.transpose` 从 `[..., PP, TP]` 维度中交换派生。`global_pcp_size` 来自 `parallel_config.prefill_context_parallel_size`。

#### 1.2.2 CP 通信流

**Prefill (PCP) 流程**（标准 FlashAttention，`attention_cp.py::_forward_prefill_cp`）：

vllm-ascend 的 PCP prefill 有**两条路径**，由 `pcp_use_hybrid_attn` 控制：

**Standard PCP**（`pcp_use_hybrid_attn=False`，`attention_cp.py:792`）：
```
Step 1: all_gather KV 跨 PCP group（dim=0）→ 每个 rank 获得完整 KV
Step 2: Q 按 head/tail 索引切分（q_head_idx / q_tail_idx），KV 按 mask/no-mask 切分
Step 3: head 部分做 attention（无 mask）、tail 部分做 attention（有 causal mask）
Step 4: _npu_attn_out_lse_update：NPU 融合 LSE merge 合并 head/tail mask/no-mask 四路输出
Step 5: index_select 按 q_full_idx 恢复原始顺序 → 最终输出
```

**Hybrid PCP**（`pcp_use_hybrid_attn=True`，`attention_cp.py:797`）：
```
Step 1: _gather_and_restore_pcp_qkv：concat(QKV) + all_gather 跨 PCP group → 恢复顺序 → 每个 rank 获得完整 QKV
Step 2: Q 按 head/tail 索引切分
Step 3: head 部分做 attention（无 mask）、tail 部分做 attention（有 causal mask）
Step 4: concat head/tail output + index_select 按 q_full_idx 恢复顺序
Step 5: all_gather attn_output 跨 PCP group + index_select 按 pcp_exit_fa_scatter_idx → 最终输出
```

> **仿真关键差异**：Standard PCP all_gather **KV**（2×num_kv_heads × head_dim），Hybrid PCP all_gather **QKV**（(num_heads + 2×num_kv_heads) × head_dim）。数据量差 ~1/3（对 GQA 模型更大）。且 Standard PCP 用 `_npu_attn_out_lse_update` 做 NPU 融合 LSE merge，Hybrid 无 LSE merge（纯 index_select 重排）。

**Decode (DCP) 流程**（标准 FlashAttention，`attention_cp.py::_forward_decode_pcp_dcp`）：

```
Step 1: all_gather query 跨 DCP group（在 head 维度，dim=1）→ 每个 rank 获得完整 query
Step 2: 各 rank 用分布式 KV cache 做 decode attention
        - fused kernel 通过 block_tables 直接读取本 rank 的 KV 分片
        - 索引: num_computed_tokens_of_pcp_dcp[:, pcp_rank, dcp_rank]
Step 3: _process_attn_out_lse: concat(attn_out, softmax_lse) + all_to_all_single (if dcp_size>1)
Step 4: _npu_attention_update: LSE 加权合并跨 PCP+DCP group → 最终输出
```

**SFA/MLA 的关键差异**：

| | 标准 FlashAttention | SFA / MLA |
|---|---|---|
| PCP KV 处理 | all_gather QKV via `_gather_and_restore_pcp_qkv`（标准 attention 专用） | 不使用 `_gather_and_restore_pcp_qkv`；通过 `gather_kv_cross_cp` / `gather_kv_cross_cp_compact` 跨 PCP+DCP 组 all_gather KV cache（SFA 使用非标准 interleaved KV 布局） |
| DCP KV 处理 | KV cache 分布式存储，不做 all_gather | 先 `gather_kv_cross_cp` 跨 DCP 组 all_gather KV cache，再做 attention |

PCP 和 DCP 可以**同时启用**（PCP 负责 prefill 阶段，DCP 负责 decode 阶段）。decode 输出通过 `_process_attn_out_lse`（concat + all_to_all_single）和 `_npu_attention_update`（LSE merge）处理；chunked prefill 上下文输出通过 `_gather_global_context_output` 和 `_update_global_context_output` 处理。

#### 1.2.3 关键通信原语

| 函数 | 作用 | 阶段 | 来源 |
|------|------|------|------|
| `get_pcp_group().all_gather(KV, dim=0)` | PCP 组内聚合完整 KV（Standard PCP 路径） | Prefill | `attention_cp.py:792` |
| `get_pcp_group().all_gather(QKV, dim=0)` | PCP 组内聚合完整 QKV（Hybrid PCP 路径，`_gather_and_restore_pcp_qkv`） | Prefill | `attention_cp.py:821` |
| `get_pcp_group().all_gather(attn_output, dim=0)` | PCP 组内聚合 attn_output（仅 Hybrid PCP 路径出口） | Prefill | `attention_cp.py:1014` |
| `get_dcp_group().all_gather(query, dim=1)` | DCP 组内在 head 维度 all_gather query | Decode | `attention_cp.py:559` |
| `_process_attn_out_lse()` | concat(out, lse) + all_to_all_single (if dcp>1) | Decode | `common_cp.py:107` |
| `dist.all_to_all_single(group=dcp_group)` | Chunked prefill 上下文输出重排 | Prefill (chunked) | `attention_cp.py: _gather_global_context_output` |

| `_npu_attn_out_lse_update()` | NPU 融合 head/tail mask/no-mask LSE merge（内部调用 `torch_npu.npu_attention_update`）— Standard PCP Prefill 路径 | Prefill | `common_cp.py:158` |
| `_update_out_and_lse()` | 纯 Python 实现的 LSE merge（无 NPU 依赖）| Prefill (chunked) / fallback | `common_cp.py:182` |

#### 1.2.4 Attention 类型适配

| 文件 | 类型 | 适用模型 | CP 差异 |
|------|------|----------|---------|
| `attention_cp.py` | 标准 Flash Attention | Llama、Qwen 等 | head/tail split + LSE merge |
| `sfa_cp.py` | Sparse Flash Attention | 使用 SFA 的模型 | 额外处理 sparse mode 参数 |
| `mla_cp.py` | Multi-head Latent Attention | DeepSeek-V2/V3 | 额外包含 `kv_b_proj` + `kv_tail_proj` 索引 |

#### 1.2.5 Chunked Prefill + CP

vllm-ascend 通过 `CPChunkedContextMetadata` 和 `AscendMetadataForPrefill.ChunkedContextMetadata` 处理 chunked prefill 与 CP 的交互。

**chunked prefill CP 上下文通信流**（`attention_cp.py::_forward_prefill_cp` 中的 `enable_chunked_prefill` 分支，L987-1007）：

```
For each chunk token:
  Step 1: 当前 chunk 的 dcp attention → dcp_output + dcp_lse
  Step 2: concat(dcp_output, dcp_lse) → local_context_output
  Step 3: _gather_global_context_output → all_to_all_single 跨 DCP group (if dcp_size>1)
  Step 4: all_gather(global_context_output, dim=0) 跨 PCP group (if pcp_size>1)
  Step 5: _update_global_context_output → LSE merge（纯 Python 实现，无 NPU 依赖）
  Step 6: _update_chunk_attn_out_lse_with_current_attn_out_lse → 更新当前 chunk 的 attn output/LSE
```

**关键仿真参数**：
```python
# chunked prefill 中，每个 PCP rank 的 chunk 长度独立追踪
local_context_lens_allranks:  # [num_reqs, pcp_size, dcp_size]
actual_seq_lengths_kv = cumsum(local_chunked_kv_lens_rank)
```

这对仿真很重要：如果 chunked prefill 启用且 CP>1，**每个 chunk 只包含该 rank 负责的 token 子集**，通信量按实际 token 数计算（`seq_len / cp_size / num_chunks`），而非完整 seq_len。chunk 边界处需要额外处理 `ChunkedContextMetadata` 的 `local_context_lens_allranks`。

## 2. ascend_msmodeling 现状分析

### 2.1 现有并行配置

`tensor_cast/model_config.py` 的 `ParallelConfig` 已有以下并行相关字段：

| 字段 | 说明 | 受 cp_size 影响 |
|------|------|:---:|
| `tensor_parallel_size` | TP size | ✅ world_size 约束 |
| `data_parallel_size` | DP size（由 world_size // tp // pp 推导，加入 CP 后还需 // cp） | ✅ 推导公式 |
| `pipeline_parallel_size` | PP size | ✅ world_size 约束 |
| `expert_parallel_size` | EP size | — (独立 MoE 约束) |
| `ulysses_size` | Ulysses 序列并行 size（默认 1） | ✅ 互斥检查 |
| `o_proj_tensor_parallel_size` / `o_proj_data_parallel_size` | o_proj 的 TP/DP 子分组 | ⚠️ 子约束需同步扩展 |
| `mlp_tensor_parallel_size` / `mlp_data_parallel_size` | MLP 的 TP/DP 子分组 | ⚠️ 子约束需同步扩展 |
| `lmhead_tensor_parallel_size` / `lmhead_data_parallel_size` | lm_head 的 TP/DP 子分组 | ⚠️ 子约束需同步扩展 |
| `moe_tensor_parallel_size` / `moe_data_parallel_size` | MoE 的 TP/DP 分组 | — (独立 MoE 约束) |

> **注意**：项目还有 `serving_cast/config.py::ParallelConfig`（字段命名不同：`tp_size`/`dp_size`/`ep_size`/`moe_tp_size`/`moe_dp_size`，**无 `pp_size`**），Unit 4 集成时需**同时扩展两份** ParallelConfig。serving_cast 的 world_size 约束不能包含 pp_size（该层无此字段）。

**`world_size` 约束（`__post_init__`）**：
```python
tp_size * dp_size * pp_size == world_size
```
注意：`ep_size` 不在基础 world_size 约束中——EP 作用于 MoE 层，沿用独立约束 `moe_tp * moe_dp * ep == world_size`。

加入 CP 后需要扩展为：
```python
tp_size * dp_size * pp_size * cp_size == world_size
```

同时需要修改 `dp_size` 的推导逻辑：
```python
if self.data_parallel_size is None:
    self.data_parallel_size = (
        self.world_size
        // self.tensor_parallel_size
        // self.pipeline_parallel_size
        // self.context_parallel_size
    )
```

> **子约束同步扩展**：`o_proj_*`、`mlp_*`、`lmhead_*` 的 `__post_init__` 约束（L307-L360）同样使用 `world_size // *_tp // pp` 模式推导 DP，加入 `cp_size` 后这些子约束均需改为 `world_size // *_tp // pp // cp`。当前 RFC 的 Unit 1 代码块仅展示了主约束，实施时需**全部更新**。

> **serving_cast 无 pp_size**：`serving_cast/config.py::ParallelConfig` 没有 `pp_size` 字段。该层的 world_size 约束为 `tp_size * dp_size * cp_size == world_size`（不含 pp），与 tensor_cast 层不同。Unit 4 集成时注意区分。

### 2.2 现有 rank 布局

`tensor_cast/parallel_group.py` 的 `initialize_model_parallel`：

```python
rank_groups_raw = all_ranks.reshape(
    -1, data_parallel_size, pipeline_parallel_size,
    expert_parallel_size, tensor_parallel_size
)  # [ExternalDP, DP, PP, EP, TP]
```

`ParallelGroupType` 枚举缺少 `CONTEXT_PARALLEL`。

### 2.3 与 Ulysses 的关系

ascend_msmodeling 已有 `ulysses_size`（`ParallelConfig`），Ulysses 和 CP 是两种**互斥**的长序列并行策略：

| | Ulysses | CP |
|------|---------|------|
| 切分维度 | head 维度 | token 维度 |
| 通信原语 | all-to-all | all-gather (PCP) / all-to-all-single (DCP) |
| 适用阶段 | prefill | prefill + decode |
| 对 attention 计算影响 | 每个 rank 计算全部 token × 部分 head | 每个 rank 计算部分 token × 全部 head |

仿真框架应支持**二选一**：`ulysses_size > 1` 时 CP 无效，反之亦然。实现时在 `ParallelConfig.__post_init__` 中增加互斥检查。

### 2.4 通信建模仿真基础设施

`serving_cast/communication.py` 已有 `Channel` 和 `CommunicationManager`，支持：
- `host2device_sync/async`
- `device2device_sync/async`

CP 仿真可直接复用 `CommunicationManager.device2device_sync(num_bytes)` 建模 all-gather / all-to-all 通信开销（`num_bytes` 由 Unit 2 的通信量公式计算）。

`tensor_cast/device.py` 中还有 `InterconnectTopology` 类（`bandwidth_bytes_ps` / `latency_s` / `comm_efficiency`），支持更精细的互联拓扑（CLOS / FULL_MESH）建模。Unit 2 的 `simulate_cp_overhead` 接收 `InterconnectTopology` 参数，但接入 serving_cast 时需要桥接：根据 topology 计算出等效的单次通信字节数，再调用 `CommunicationManager.device2device_sync`。

> **注意**：`InterconnectTopology.bandwidth_bytes_ps` 字段名表示 bytes/s，注释误标为 "GB/s"。

> **注意**：`InterconnectTopology` 当前仅支持 CLOS 和 FULL_MESH 拓扑（有 TODO 标注），CP 通信建模暂用固定带宽；后续可扩展 ring/torus 等拓扑类型。

## 3. Work Unit 分解

### Unit 1: 扩展 ParallelConfig 和 ParallelGroup 支持 CP

| 字段 | 内容 |

**实施要点**：

1. `ParallelConfig` 新增字段：
```python
context_parallel_size: int = 1       # CP size（含 PCP 和 DCP 的统一 group）
prefill_context_parallel_size: int = 1  # 可选的 PCP size（≤ context_parallel_size）
decode_context_parallel_size: int = 1   # 可选的 DCP size（≤ context_parallel_size）
```

2. `__post_init__` 修改 world_size 约束和 dp_size 推导：
```python
def __post_init__(self):
    if self.context_parallel_size <= 0:
        raise ValueError("context_parallel_size must be >= 1")
    if self.ulysses_size > 1 and self.context_parallel_size > 1:
        raise ValueError(
            "Ulysses (ulysses_size) and Context Parallelism (cp_size) "
            "are mutually exclusive — choose one."
        )
    # 扩展 world_size 约束
    if self.tensor_parallel_size * self.data_parallel_size \
       * self.pipeline_parallel_size * self.context_parallel_size \
       != self.world_size:
        raise ValueError(...)
```

3. `ParallelGroupType` 新增：
```python
CONTEXT_PARALLEL = "context_parallel"
```

4. rank 网格改为（在 EP 之前插入 CP）：
```python
rank_groups_raw = all_ranks.reshape(
    -1, dp_size, pp_size, cp_size, ep_size, tp_size
)
```

5. `ParallelGroupManager` 新增 `self.cp_group` 成员（**只有这一个 CP group**，没有 `o_proj_cp_group` 或 `mlp_cp_group`——CP 仅作用于 attention 计算，不切分 Linear/proj/MLP 权重）。

### Unit 2: CP 通信建模仿真

| 字段 | 内容 |
|------|------|
| **id** | `unit-002-cp-comm-modeling` |
| **depends_on** | `unit-001` |
| **scope** | 新增 `tensor_cast/cp_sim.py` |
| **acceptance_tests** | all_gather 开销随 cp_size 和 seq_len 线性增长；PCP 和 DCP 的通信量不同；chunked prefill 下通信量按实际 token 数计算 |
| **risk_level** | Tier 2 |
| **rollback_plan** | 删除 `cp_sim.py` |

**实施要点**：

```python
@dataclass
class CPSimResult:
    """CP 仿真结果"""
    pcp_comm_time_s: float   # PCP 通信耗时
    pcp_comp_time_s: float   # PCP 计算耗时（attention_update merge）
    dcp_comm_time_s: float   # DCP 通信耗时
    dcp_comp_time_s: float   # DCP 计算耗时

def simulate_cp_overhead(
    seq_len: int,
    query_len: int,
    head_dim: int,
    num_heads: int,
    num_kv_heads: int,
    cp_size: int,
    dcp_size: int,
    kv_cache_dtype: torch.dtype,
    attn_type: str,             # "flash_attn" | "sfa" | "mla"
    cp_engine: str,             # "vllm_ascend" | "sglang" | "custom"
    interconnect: InterconnectTopology,
    is_chunked_prefill: bool = False,
    pcp_use_hybrid_attn: bool = True,  # vllm-ascend PCP 模式开关
) -> CPSimResult:
    """
    模拟 CP 通信 + 计算开销。

    通信建模（vllm-ascend）：
      - Standard PCP（pcp_use_hybrid_attn=False）:
        * all_gather(KV, dim=0) 跨 PCP group
          → 数据量 = seq_len * 2 * num_kv_heads * head_dim * bytes_per_elem
        * 无后续 all_gather attn_out（输出由 index_select 重排）
        * LSE merge: _npu_attn_out_lse_update (NPU 融合)
      - Hybrid PCP（pcp_use_hybrid_attn=True）:
        * all_gather(QKV, dim=0) 跨 PCP group（_gather_and_restore_pcp_qkv）
          → 数据量 = seq_len * (num_heads + 2*num_kv_heads) * head_dim * bytes_per_elem
        * all_gather(attn_output, dim=0) 跨 PCP group
          → 单 rank 数据量 = seq_len/pcp_size * num_heads * head_dim * bytes_per_elem
        * 无 LSE merge（纯 index_select 重排）
      - DCP: all_gather(query, dim=1) 跨 DCP group（head 维度）
        * 数据量 = batch_size * num_heads * head_dim * dcp_size * float16
      - SFA/MLA 额外开销: gather_kv_cross_cp all_gather KV cache 跨 PCP+DCP group
        * 数据量 = seq_len * num_kv_heads * head_dim * 2 * kv_cache_dtype.itemsize * cp_size
        * 仅 SFA/MLA；标准 attention DCP 不做 KV all_gather（分布式 KV cache 直读）

    计算建模：
      - Standard PCP: _npu_attn_out_lse_update (NPU 融合 LSE merge，O(heads × seq_len))
      - Hybrid PCP: 无 LSE merge 开销
      - DCP (decode): _npu_attention_update (NPU 融合 LSE merge, O(heads × dcp_size))
      - sglang/custom: 使用 Python 实现的 LSE merge

    MLA 附加开销：
      - kv_b_proj: 额外的投影计算
      - kv_tail_proj: tail 部分 KV 的额外索引操作
    """
```

**三种引擎的 CP 通信模式差异**：

| 引擎 | PCP 通信 | PCP 通信（Hybrid） | DCP 通信 | LSE merge |
|------|----------|-------------------|----------|-----------|
| `vllm_ascend` | all_gather KV + `_npu_attn_out_lse_update` | all_gather QKV + all_gather attn_out (无 LSE merge) | all_gather query + fused attn on distributed KV (DCP) | `_npu_attention_update` (decode) / `_npu_attn_out_lse_update` (prefill standard) |
| `sglang` | ring send/recv KV chunk | — | 不支持（或仅 PCP） | softmax recomputation |
| `custom` | 用户指定 | 用户指定 | 用户指定 |

**Chunked Prefill 下的特殊处理**：

当 `is_chunked_prefill=True` 且 `cp_size > 1` 时：
- 每个 chunk 的 token 数 = `seq_len / cp_size / num_chunks`
- all_gather 的数据量基于该 chunk 的实际 token 数，而非完整 seq_len
- chunk 边界处需要额外处理 `ChunkedContextMetadata` 的 `local_context_lens_allranks`

**配置类设计**：

```python
@dataclass
class CPConfig:
    """Context Parallelism 仿真配置"""
    cp_size: int = 1
    cp_engine: str = "vllm_ascend"        # "vllm_ascend" | "sglang" | "custom"
    attn_type: str = "flash_attn"         # "flash_attn" | "sfa" | "mla"
    cp_kv_cache_interleave_size: int = 128
    enable_pcp: bool = True
    enable_dcp: bool = True
    
    # 引擎特定参数（不干预默认行为，仅 override）
    engine_params: dict = field(default_factory=dict)
    
    def __post_init__(self):
        if self.cp_engine not in ("vllm_ascend", "sglang", "custom"):
            raise ValueError(f"Unknown cp_engine: {self.cp_engine}")
        if self.attn_type not in ("flash_attn", "sfa", "mla"):
            raise ValueError(f"Unknown attn_type: {self.attn_type}")
```

**引擎切换逻辑**：

```python
class CPEngine(ABC):
    """CP 引擎抽象基类"""
    
    @abstractmethod
    def get_pcp_comm_ops(self, params: CPCommParams) -> list[CommOp]:
        """返回 PCP 阶段的通信算子序列"""
        ...
    
    @abstractmethod
    def get_dcp_comm_ops(self, params: CPCommParams) -> list[CommOp]:
        """返回 DCP 阶段的通信算子序列"""
        ...
    
    @abstractmethod
    def get_merge_strategy(self) -> str:
        """返回 LSE merge 策略: "npu_fusion" | "softmax_recompute" | "custom" """
        ...

class VllmAscendCPEngine(CPEngine):
    """
    vllm-ascend 风格：
    PCP: all_gather QKV → head/tail split attn → all_gather out+LSE → _npu_attention_update
    DCP: all_gather query → fused attn on distributed KV → _process_attn_out_lse → _npu_attention_update
    """
    ...

class SglangCPEngine(CPEngine):
    """
    sglang 风格：
    PCP: ring-based KV chunk 传递 → 独立 attention → softmax recomputation merge
    DCP: 通常不支持
    """
    ...
```

## 7. 开放问题

1. **PD 分离**：vllm-ascend 支持 Prefill/Decode 分离部署（`kv_transfer` 子系统），CP 在 PD 场景下 KV cache 需要跨 P 和 D 节点传输。当前 RFC 暂不覆盖 PD 分离场景，留待后续 RFC。

2. **CP + EP 组合**：vllm-ascend 的实际 rank 网格不含 EP 维度（EP 通过 transpose 派生）。ascend_msmodeling 是否需要在 rank 布局中同时包含 CP 和 EP，取决于是否计划同时使用 MoE 和长序列。

3. **真实数据校准**：仿真精度依赖 `InterconnectTopology` 的带宽参数和 `simulate_cp_overhead` 的计算模型。建议后续用 vllm-ascend 真实 profiling 数据校准。

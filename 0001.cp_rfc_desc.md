# RFC 0001 白话解读：Context Parallelism 仿真支持

> 本文是对 `0001.cp_rfc.md` 的逐点白话解读，面向对分布式推理不太熟悉的小白读者。
> 读完本文，你不需要有任何并行计算背景，只需要知道"大模型推理就是把一段文字扔给 AI，AI 逐字回复"。

---

## 序言：为什么需要这篇文章？

RFC（Request for Comments）是工程师用来描述"我打算做什么、怎么做"的技术文档。`0001.cp_rfc.md` 讲的是如何在 `ascend_msmodeling`（一个推理仿真框架）里加入 **Context Parallelism（上下文并行）** 的支持。

原 RFC 写得很技术化，假设读者已经了解很多背景知识。本文的任务就是把每一个概念、每一步设计都拆开揉碎了讲清楚——就像教一个刚入职的新人理解整个系统那样。

## 第零章：先搞懂几个核心概念

在深入 RFC 之前，我们需要先理解几个贯穿全文的"行话"。

### 0.1 什么是「推理」（Inference）？

大模型（如 ChatGPT、Claude）的工作分两阶段：
- **训练**：给模型"喂"海量文本，让它学会语言规律。这需要几千张 GPU 跑几个月。
- **推理**：模型已经训练好了，用户发一句话过来，模型逐字生成回复。这是我们日常使用大模型的场景。

**ascend_msmodeling 是干什么的？** 它是一个**仿真框架**——不真正跑模型，而是用数学公式估算：给定模型大小、序列长度、GPU 数量等参数，推理需要多长时间。就像一个"推理性能计算器"。

### 0.2 为什么需要并行？

一个 70B（700亿参数）的模型，单张 GPU 的显存放不下。所以需要**多张 GPU 协同工作**，这就是并行。

常见的并行策略就像工厂分工：

| 并行策略 | 类比 | 一句话解释 |
|----------|------|-----------|
| **TP（张量并行）** | 一个零件由 4 个人各拧一部分螺丝 | 把模型的一层切分到多张卡上同时计算 |
| **PP（流水线并行）** | 流水线上工序 1→2→3，每人做一段 | 把模型的不同层放到不同的卡上 |
| **DP（数据并行）** | 分 4 条一样的流水线，各处理一批订单 | 多张卡各跑一份完整模型，处理不同的请求 |
| **EP（专家并行）** | 不同问题的专家坐不同窗口 | MoE 模型的"专家"分布在不同卡上 |
| **CP（上下文并行）** | 一本长书撕成几份，每人读一段 | 把长序列切成多段，每张卡只存一部分 |

### 0.3 什么是 Context（上下文）？什么是 KV Cache？

当用户发送一段文字给大模型时，这段文字就是**上下文（context）**。

模型处理上下文时，会为每个 token（可以理解为"每个字"）计算两部分数据：
- **K（Key）**：可以理解为这个 token 的"标签"
- **V（Value）**：可以理解为这个 token 的"内容"

K 和 V 会被存起来，这样生成下一个 token 时不用从头算起。这个存储就叫 **KV Cache**。

**问题来了**：如果上下文非常长（比如一本书），KV Cache 可能大到单张 GPU 装不下。这就是 CP 要解决的问题。

### 0.4 什么是 Attention（注意力机制）？

Attention 是大模型的核心机制。简单来说：

> 模型在生成下一个字时，会"回顾"之前的所有字，判断哪些字最重要，然后根据这些重要信息来决定下一个字是什么。

```
用户输入："今天天气真"
模型回想："今天"（普通）、"天气"（重要！）、"真"（最近）
→ 模型决定下一个字："好" → "今天天气真好"
```

Attention 的计算需要完整的 KV Cache。CP 把 KV Cache 分散存储，所以需要额外的通信来"凑齐"信息。

### 0.5 Prefill 和 Decode：推理的两个阶段

大模型推理分两个阶段：

```
┌─────────────────────────────────────────────────────┐
│  用户输入："请用三句话介绍人工智能"                  │
│                                                     │
│  Phase 1: Prefill（预填充）                          │
│  ├─ 把整个输入句子一次性处理                          │
│  ├─ 计算所有输入 token 的 KV Cache                   │
│  └─ 并行计算，速度很快（受算力限制）                  │
│                                                     │
│  Phase 2: Decode（解码）                              │
│  ├─ 逐字生成回复："人"→"工"→"智"→"能"→"是"→...      │
│  ├─ 每次只生成一个字，但需要查看所有 KV Cache         │
│  └─ 串行生成，速度很慢（受内存带宽限制）              │
└─────────────────────────────────────────────────────┘
```

**关键区别**：Prefill 一次处理很多字（算力密集型），Decode 一次只处理一个字（内存密集型）。CP 在这两个阶段使用不同的策略。

### 0.6 集合通信原语：GPU 之间怎么传数据？

多张 GPU 之间需要传数据，这用到了"集合通信"：

| 通信原语 | 做什么 | 类比 |
|----------|--------|------|
| **all_gather** | 每张卡把自己的数据广播给组内所有卡，每张卡最后都有完整数据 | 小组每人写一段笔记，然后复印给所有人 |
| **all_to_all** | 每张卡把数据分发给组内其他卡，同时也接收别人的数据 | 小组每人把不同科目作业交给对应的课代表 |
| **all_reduce** | 每张卡的数据加在一起，结果广播给所有卡 | 小组统计总销售额，每人知道最终数字 |

CP 主要用 `all_gather`（收集数据）和 `all_to_all_single`（重排数据）。

### 0.7 什么是 LSE（Log-Sum-Exp）？为什么需要 Merge？

在 Attention 计算中，Softmax 的结果需要用 LSE 来稳定数值。当序列被 CP 切分后，每张卡独立计算 attention，得到**局部的** attention 输出和 LSE。最终需要把这些局部结果按 LSE 加权合并（merge）成**全局的**输出。

```
局部结果 A: output_A, lse_A
局部结果 B: output_B, lse_B
全局结果 = (output_A × exp(lse_A) + output_B × exp(lse_B)) / exp(全局_lse)
```

这就是 LSE merge。不同的 attention 类型有不同的 merge 实现方式（有的用 NPU 硬件加速，有的用纯 Python）。

### 0.8 NPU 是什么？

NPU（Neural Processing Unit）是华为昇腾系列 AI 芯片。`torch_npu` 是其对应的 PyTorch 扩展库。vllm-ascend 是专门为昇腾 NPU 优化的推理引擎。一些 CP 操作（如 LSE merge）有 NPU 专用的硬件加速版本。

## 第一章：背景与动机（RFC §1）

### 1.1 CP 并行概述 — RFC §1.1

**RFC 原文说的是什么**：

Context Parallelism 是一种专门处理长序列的并行策略。当序列太长、KV Cache 太大，单卡放不下时，CP 把序列切成多段分到不同卡上。计算 attention 时，通过集合通信把分散的 KV 拼回来再算。

vllm-ascend 的 CP 分两层：
- **PCP**（Prefill Context Parallelism）：处理 prefill 阶段的长序列
- **DCP**（Decode Context Parallelism）：处理 decode 阶段的分布式 KV Cache

两者使用**独立的通信组**（`get_pcp_group()` 和 `get_dcp_group()`），这意味着可以只开 PCP、只开 DCP，或同时开。

**补充上下文**：

PCP 和 DCP 可以设置不同的并行度。比如 prefill 阶段用 4 张卡（PCP=4），但 decode 阶段可能只需要 2 张卡（DCP=2）。这是因为 prefill 需要处理大量 token（算力要求高），而 decode 每次只处理一个 token（算力要求低，但 KV Cache 大）。

**为什么 ascend_msmodeling 需要这个**：

当前 ascend_msmodeling 已经支持 TP、DP、EP、PP 和 Ulysses 序列并行，但缺少 CP。加入 CP 后，就可以仿真 "长序列 + 多卡推理" 场景的性能了。

### 1.2 参考实现：vllm-ascend CP 架构 — RFC §1.2

#### 1.2.1 Rank 布局 — RFC §1.2.1

**RFC 原文说的是什么**：

vllm-ascend 把 `world_size`（总 GPU 数量）张卡排成一个 5 维的"网格"：

```
[ExternalDP, DP, PP, PCP, TP]
```

每个维度代表一种并行策略的大小。实际 rank（GPU 编号）通过 reshape 分配到这个网格中。

**补充上下文**：

- "rank" 就是 GPU 的编号（第 0 号卡、第 1 号卡...）
- "world_size" = 总共多少张 GPU
- 网格的每个维度的乘积 = world_size（即 `ExternalDP × DP × PP × PCP × TP = world_size`）

**重要细节**：vllm-ascend 的 rank 网格**没有独立的 EP 维度**。当启用 EP 时，EP 分组是通过交换（transpose）TP 和 PP 维度派生出来的，不是直接多一个维度。这是一个重要的架构细节——ascend_msmodeling 如果想兼容 vllm-ascend 的行为，需要理解这一点。

> **🤔 小白问题：为什么要用网格排列 GPU？**
>
> 就像教室里的座位安排——第 1 排第 3 列的同学属于第 1 排（PP=第几排），也属于第 3 列（TP=第几列）。网格让你一眼看出谁和谁是"同组"的。

#### 1.2.2 CP 通信流 — RFC §1.2.2

这是 RFC 最核心的部分，我们分开细讲。

---

##### PCP（Prefill CP）的 Standard 路径

**RFC 原文说的流程**（`pcp_use_hybrid_attn=False`，默认路径）：

```
Step 1: all_gather KV 跨 PCP group（dim=0 维度）
        → 每张卡拿到完整 KV
Step 2: Q 按 head/tail 索引切分，KV 按 mask/no-mask 切分
Step 3: head 部分做 attention（无 mask）、tail 部分做 attention（有 causal mask）
Step 4: _npu_attn_out_lse_update：NPU 融合 LSE merge 合并 head/tail mask/no-mask 四路输出
Step 5: index_select 按 q_full_idx 恢复原始顺序 → 最终输出
```

**白话解读**：

1. **all_gather KV**：当前每张 GPU 上只有一部分 KV，通过 all_gather 让所有卡都拿到完整 KV。这是最大的通信开销。
   - 注意：Standard 路径**只** all_gather KV，**不** all_gather Q！因为 Q 在每张卡上是完整的（不需要通信）。
   - 通信量 = `seq_len × 2 × num_kv_heads × head_dim × 2字节（fp16）`

2. **Q 切分 head/tail**：Q 是"查询"，按 prefill 的需要切成"头"和"尾"两部分。这是 vllm-ascend 的 Ring Flash Attention 实现细节。
   - "head" 部分是序列的前半段，"tail" 部分是后半段

3. **分开做 attention**：
   - Head 部分不需要 causal mask（因为它能看到全部上下文）
   - Tail 部分需要 causal mask（后面的词不能"偷看"前面的词）

4. **LSE merge**：`_npu_attn_out_lse_update`（common_cp.py:158）把 head/tail、mask/no-mask 的四路 attention 输出按 LSE 合并。这是**NPU 硬件加速**版本——只在昇腾芯片上有效。

5. **index_select 恢复顺序**：因为 Q 被切分过，最后需要用预计算的索引恢复原始 token 顺序。

---

##### PCP（Prefill CP）的 Hybrid 路径

**RFC 原文说的流程**（`pcp_use_hybrid_attn=True`）：

```
Step 1: _gather_and_restore_pcp_qkv：concat(QKV) + all_gather 跨 PCP group → 每张卡获得完整 QKV
Step 2: Q 按 head/tail 索引切分
Step 3: head/tail 分别做 attention
Step 4: concat head/tail output + index_select 按 q_full_idx 恢复顺序
Step 5: all_gather attn_output 跨 PCP group + index_select 按 pcp_exit_fa_scatter_idx → 最终输出
```

**白话解读**：

1. **all_gather QKV**：和 Standard 不同，Hybrid 把 Q、K、V **全部** all_gather。通信量更大（多了 Q）。
   - 通信量 = `seq_len × (num_heads + 2 × num_kv_heads) × head_dim × 2字节`
   - 对于 GQA 模型（num_kv_heads < num_heads），Hybrid 比 Standard 多传输约 30-50% 数据

2. **切分 + attention**：和 Standard 一样，Q 切 head/tail 分开算。

3. **无 LSE merge**：Hybrid 路径**不**使用 `_npu_attention_update` 也不使用 `_update_out_and_lse`。因为每张卡已经有完整 QKV，输出只需 index_select 重排就行。

4. **all_gather attn_output**：最终输出需要 all_gather 一次（L1014），这是第二轮通信。

**Standard vs Hybrid 总结**：

| 对比项 | Standard PCP | Hybrid PCP |
|--------|-------------|------------|
| 第一轮通信 | all_gather **KV**（小） | all_gather **QKV**（大） |
| 第二轮通信 | 无 | all_gather **attn_output** |
| LSE merge | 有（NPU 融合） | 无（index_select 重排） |
| 适用场景 | 默认，通信量小 | 某些情况下数值精度更好 |

> **补充细节（来自 CP 参考笔记）**：Hybrid 路径仅在标准 FlashAttention 中可用（attention_cp.py:786-803 判断）。SFA 和 MLA 不使用这个路径——它们有自己的 `gather_kv_cross_cp` 机制。

---

##### DCP（Decode CP）流程

**RFC 原文说的流程**：

```
Step 1: all_gather query 跨 DCP group（head 维度，dim=1）
        → 每张卡拿到完整 query
Step 2: 各 rank 用分布式 KV cache 做 decode attention
        - fused kernel 通过 block_tables 直接读本 rank 的 KV 分片
Step 3: _process_attn_out_lse: concat(attn_out, softmax_lse)
        + all_to_all_single（仅 dcp_size>1 时）
Step 4: _npu_attention_update: LSE 加权合并跨 PCP+DCP group → 最终输出
```

**白话解读**：

1. **all_gather query**：decode 阶段每次都只处理一个新 token（当前要生成的字），所以 query 很小。all_gather 的是 head 维度（dim=1），不是 token 维度（dim=0）。
   - 通信量 = `batch_size × num_heads × head_dim × dcp_size × 2字节`
   - 很小！因为 decode 阶段 token 只有 1 个

2. **分布式 KV 读取**：每张卡只读取自己负责的那段 KV Cache。通过 block_tables 索引直接定位。**不需要** all_gather KV（这是和 prefill 最大的不同）。

3. **_process_attn_out_lse**（common_cp.py:107）：这是 decode 路径的关键函数。它做两件事：
   - 把 attention 输出和 softmax LSE 拼接（concat）
   - 如果 dcp_size > 1，通过 all_to_all_single 跨 DCP 组重排

4. **_npu_attention_update**（common_cp.py:129）：NPU 硬件加速的 LSE merge。**仅在 decode 路径调用**（attention_cp.py:637、mla_cp.py:745），对应的 prefill 版本是 `_npu_attn_out_lse_update`。

> **重要纠正（来自 v2.3 审查）**：`_npu_attention_update` 是 **decode 专用** 的，不是 prefill+decode 双阶段。prefill 阶段用 `_npu_attn_out_lse_update`（Standard PCP 路径）。`_process_attn_out_lse` 是真实存在的函数（common_cp.py:107），之前版本错误地认为它不存在。

##### SFA 和 MLA 的差异 — RFC §1.2.2 表格

**RFC 原文说的**：

| | 标准 FlashAttention | SFA / MLA |
|---|---|---|
| PCP KV 处理 | all_gather QKV（`_gather_and_restore_pcp_qkv`） | 不调用 `_gather_and_restore_pcp_qkv`；用 `gather_kv_cross_cp` all_gather KV cache |
| DCP KV 处理 | KV cache 分布式存储，不做 all_gather | 先 `gather_kv_cross_cp` all_gather KV，再做 attention |

**白话解读**：

- **SFA（Sparse Flash Attention）**：一种更高效的 attention 算法。它在 CP 场景下需要额外的 KV 收集步骤（`gather_kv_cross_cp`），即使 decode 阶段也要 all_gather KV cache。
- **MLA（Multi-head Latent Attention）**：DeepSeek-V2/V3 使用的 attention 类型，用低秩压缩减少 KV Cache 大小。也有额外的 KV 投影和索引操作。

**核心区别**：标准 Flash Attention 在 decode 时**不需要** all_gather KV（每张卡直接读自己的 KV 分片）。但 SFA/MLA 在 decode 时**需要**先 all_gather KV，再算 attention。这是仿真建模的重要差异——SFA/MLA 的 decode 通信开销比标准 attention 大得多。

##### Chunked Prefill + CP — RFC §1.2.5

**RFC 原文说的是什么**：

Chunked Prefill 是另一种优化：把长 prefill 序列切成多个 chunk（小块），分批处理，减少延迟。当 chunked prefill 和 CP 同时启用时，每个 chunk 只包含该 rank 负责的 token 子集。

**白话解读**：

```
假设：一个 128K 序列，CPU=4，分 4 个 chunk

原始序列：[T0, T1, T2, T3, ..., T127999]（128000 tokens）

CP=4 切分：
  Rank 0: [T0, T4, T8, ...]（32000 tokens，交错切分）
  Rank 1: [T1, T5, T9, ...]
  Rank 2: [T2, T6, T10, ...]
  Rank 3: [T3, T7, T11, ...]

再 Chunk=4：
  Rank 0 的每个 chunk: 32000/4 = 8000 tokens

所以每次 all_gather 的数据量按 8000 tokens 算，不是完整的 32000！
```

**chunked prefill CP 通信流**（6 步）：

```
Step 1: 当前 chunk 做 dcp attention → dcp_output + dcp_lse
Step 2: concat(output, lse) → local_context_output
Step 3: _gather_global_context_output → all_to_all_single 跨 DCP group
Step 4: all_gather(global_context_output, dim=0) 跨 PCP group
        → 使用单独通信流 (cp_chunkedprefill_comm_stream) 实现通信计算重叠
Step 5: _update_global_context_output → LSE merge（纯 Python，无 NPU 加速）
Step 6: _update_chunk_attn_out_lse_with_current_attn_out_lse
```

**关键仿真参数**：`local_context_lens_allranks: [num_reqs, pcp_size, dcp_size]`——这是一个 3D 数组，追踪每个请求在每个 PCP rank、每个 DCP rank 上的 chunk 长度。

> **为什么 chunked prefill 的 LSE merge 不用 NPU 加速？** 因为 chunk 大小是动态变化的，NPU 融合 kernel 需要固定大小的输入。所以这里用纯 Python 的 `_update_out_and_lse()`（common_cp.py:182）。

#### 1.2.3 关键通信原语汇总 — RFC §1.2.3

这是 RFC 中的通信原语表格，我加上白话解释：

| 函数 | 做什么（白话） | 谁在用 | 在哪（源码位置） |
|------|---------------|--------|-----------------|
| `get_pcp_group().all_gather(KV, dim=0)` | 把 KV 从各卡收集起来（Standard PCP） | Prefill | attention_cp.py:792 |
| `get_pcp_group().all_gather(QKV, dim=0)` | 把 Q、K、V 全部收集起来（Hybrid PCP） | Prefill | attention_cp.py:821 |
| `get_pcp_group().all_gather(attn_output, dim=0)` | 把 attention 输出收集起来（仅 Hybrid PCP 出口） | Prefill | attention_cp.py:1014 |
| `get_dcp_group().all_gather(query, dim=1)` | 在 head 维度上收集 query | Decode | attention_cp.py:559 |
| `_process_attn_out_lse()` | 输出+LSE 拼接，加 all_to_all_single 重排 | Decode | common_cp.py:107 |
| `dist.all_to_all_single(group=dcp_group)` | DCP 组内数据重排（chunked prefill 上下文） | Prefill (chunked) | attention_cp.py 中的 `_gather_global_context_output` |
| `gather_kv_cross_cp()` | SFA 专用：跨 PCP+DCP 组收集 KV cache | Prefill / Decode | sfa_cp.py:373 |
| `_npu_attention_update()` | NPU 硬件加速 LSE merge（**仅 Decode**） | Decode | common_cp.py:129 |
| `_npu_attn_out_lse_update()` | NPU 硬件加速 LSE merge（**Standard PCP Prefill**） | Prefill | common_cp.py:158 |
| `_update_out_and_lse()` | 纯 Python LSE merge（无 NPU 依赖） | Prefill (chunked) / fallback | common_cp.py:182 |

> **关于 LSE merge 函数的重要区分**：
> - `_npu_attention_update` ≠ `_npu_attn_out_lse_update`——名字很像但完全不同！前者是 decode 用的，后者是 prefill Standard PCP 用的
> - `_update_out_and_lse` 是纯 Python 实现，用于 chunked prefill 等不需要 NPU 加速的场景
> - `_process_attn_out_lse` 是做 all_to_all 通信的，本身不做 LSE merge——它是 merge 之前的准备步骤

## 第二章：ascend_msmodeling 现状分析（RFC §2）

### 2.1 现有并行配置 — RFC §2.1

**RFC 原文说的**：`tensor_cast/model_config.py` 的 `ParallelConfig` 已有 TP、DP、PP、EP、Ulysses 等并行字段。加入 CP 后需要修改 world_size 约束和 dp_size 推导。

**白话解读**：

当前 ParallelConfig 的核心约束：

```python
tp_size × dp_size × pp_size == world_size  # 总 GPU 数必须刚好分完
```

加入 CP 后变成：

```python
tp_size × dp_size × pp_size × cp_size == world_size
```

**dp_size 自动推导也要改**：

```python
# 之前
dp_size = world_size // tp_size // pp_size

# 之后
dp_size = world_size // tp_size // pp_size // cp_size
```

**为什么要改子约束？** 除了主约束，还有以下子约束也需要同步修改：

- `o_proj`（输出投影层）的子分组约束
- `mlp`（前馈网络层）的子分组约束
- `lm_head`（输出层）的子分组约束

这些子约束都使用 `world_size // *_tp // pp` 模式推导 DP，加入 CP 后需要全部改为 `world_size // *_tp // pp // cp`。

> **⚠️ 重要：项目有两个 ParallelConfig！**
> 
> 一个是 `tensor_cast/model_config.py`（字段名较长：`tensor_parallel_size`、`data_parallel_size` 等）
> 另一个是 `serving_cast/config.py`（字段名较短：`tp_size`、`dp_size` 等，**且没有 pp_size 字段**）
> 
> Unit 4 集成时需要**同时修改两份** ParallelConfig，且 serving_cast 的约束是 `tp × dp × cp == world_size`（不含 pp），与 tensor_cast 不同。

### 2.2 现有 rank 布局 — RFC §2.2

**原文**：当前 rank 网格是 `[ExternalDP, DP, PP, EP, TP]`，没有 CP 维度和 CONTEXT_PARALLEL 枚举。

**白话**：rank 网格就像一个多维表格，把 GPU 编号分配到各个并行策略维度上。目前是 5 维，需要加入 CP 变成 6 维：

```python
# 之前
rank_groups = all_ranks.reshape(-1, dp, pp, ep, tp)  # 5 维

# 之后（CP 插在 EP 之前）
rank_groups = all_ranks.reshape(-1, dp, pp, cp, ep, tp)  # 6 维
```

### 2.3 与 Ulysses 的关系 — RFC §2.3

**原文**：Ulysses 和 CP 是两种**互斥**的长序列并行策略。仿真框架应支持二选一。

**白话**：

| 对比项 | Ulysses | CP |
|--------|---------|-----|
| 切分什么 | head 维度（把 attention head 分给不同卡） | token 维度（把序列切给不同卡） |
| 通信方式 | all-to-all | all-gather / all-to-all-single |
| 适用场景 | 适合 attention head 多的模型 | 适合序列特别长的场景 |

**为什么互斥？** 两种策略都在"切分 attention 计算"，但切的方式不同、通信方式不同。同时用会冲突。所以在代码里加检查：如果 `ulysses_size > 1` 且 `cp_size > 1`，直接报错。

### 2.4 通信建模仿真基础设施 — RFC §2.4

**原文**：`serving_cast/communication.py` 已有 `CommunicationManager.device2device_sync(num_bytes)` 可以用。`tensor_cast/device.py` 还有 `InterconnectTopology` 可以做更精细的带宽/延迟建模。

**白话**：

当前框架有两种通信建模方式：
1. **简单方式**：`CommunicationManager.device2device_sync(num_bytes)`——给它数据量，它返回耗时。适合快速估算。
2. **精细方式**：`InterconnectTopology`——可以指定 GPU 互联类型（CLOS 交换机 / 全互联），有带宽（bandwidth_bytes_ps）和延迟（latency_s）参数。适合精确仿真。

CP 仿真需要用"精细方式"建模，因为 CP 的通信模式在不同 GPU 拓扑下表现差异很大。

> **注意**：`bandwidth_bytes_ps` 的注释误标为 "GB/s"，实际字段名是 "bytes/s"。这是一个已知 bug。

## 第三章：Work Unit 分解（RFC §3）

RFC 把 CP 支持拆成 5 个 "Work Unit"（工作单元），每个 Unit 可独立开发、测试、回滚。

### Unit 1：扩展 ParallelConfig 和 ParallelGroup（RFC §3 Unit 1）

**做什么**：在 ascend_msmodeling 的配置系统里加上 CP 相关的字段和约束。

**具体要改的代码**：

1. **`ParallelConfig` 新增 3 个字段**：
```python
context_parallel_size: int = 1        # 总的 CP size
prefill_context_parallel_size: int = 1  # PCP size（可小于总 size）
decode_context_parallel_size: int = 1   # DCP size（可小于总 size）
```

2. **`__post_init__` 新增检查和约束**：
   - `ulysses_size > 1` 时不允许 `cp_size > 1`（互斥）
   - world_size 约束加入 cp_size 因子
   - dp_size 自动推导加入 cp_size 因子

3. **`ParallelGroupType` 新增枚举**：`CONTEXT_PARALLEL = "context_parallel"`

4. **rank 网格插入 CP 维度**（在 EP 之前）

5. **`ParallelGroupManager` 新增 `self.cp_group`**：只有**一个** CP 通信组。注意：没有 `o_proj_cp_group` 或 `mlp_cp_group`——CP 只影响 attention 计算，不影响 Linear/projection/MLP 层的权重切分。

**验收标准**：CP group 的 world_size == cp_size；每个 rank 正确归入对应 CP group；Ulysses 和 CP 同时启用时报错。

### Unit 2：CP 通信建模仿真（RFC §3 Unit 2）

**做什么**：新增 `tensor_cast/cp_sim.py`，实现 `simulate_cp_overhead()` 函数，根据输入的模型参数、CP 配置、序列长度，估算 CP 带来的通信和计算开销。

**核心数据结构**：
```python
@dataclass
class CPSimResult:
    pcp_comm_time_s: float   # PCP 通信耗时（秒）
    pcp_comp_time_s: float   # PCP 计算耗时（秒）
    dcp_comm_time_s: float   # DCP 通信耗时（秒）
    dcp_comp_time_s: float   # DCP 计算耗时（秒）
```

**通信量公式（白话版）**：

| 场景 | 通信量计算 |
|------|-----------|
| Standard PCP all_gather KV | `seq_len × 2 × num_kv_heads × head_dim × 2字节` |
| Hybrid PCP all_gather QKV | `seq_len × (num_heads + 2×num_kv_heads) × head_dim × 2字节` |
| Hybrid PCP all_gather attn_output | `(seq_len/pcp_size) × num_heads × head_dim × 2字节` |
| DCP all_gather query | `batch_size × num_heads × head_dim × dcp_size × 2字节` |
| SFA/MLA gather_kv_cross_cp | `seq_len × num_kv_heads × head_dim × 2 × kv_cache_dtype × cp_size` |

**计算开销**：
- Standard PCP LSE merge：`O(num_heads × seq_len)`，NPU 加速
- Hybrid PCP：无 LSE merge 开销
- DCP LSE merge：`O(num_heads × dcp_size)`，NPU 加速
- Chunked Prefill：纯 Python LSE merge，`O(num_heads × chunk_size)`

**三种引擎的差异**：

| 引擎 | PCP 通信 | DCP 通信 | LSE merge |
|------|----------|----------|-----------|
| vllm_ascend | all_gather KV 或 QKV | all_gather query + distributed KV | NPU 融合加速 |
| sglang | ring 式 KV chunk 传递 | 通常不支持 | softmax 重算 |
| custom | 用户自定义 | 用户自定义 | 用户自定义 |

**Chunked Prefill 的特殊处理**：当启用 chunked prefill 时，每个 chunk 的 token 数 = `seq_len / cp_size / num_chunks`，通信量按 chunk 大小计算，不是完整序列长度。

### Unit 3：CP 仿真配置系统（RFC §3 Unit 3）

**做什么**：设计 `CPConfig` 类和 `CPEngine` 抽象，让用户可以方便切换不同的 CP 实现（vllm-ascend 风格、sglang 风格、自定义风格）。

**配置类设计**：
```python
@dataclass
class CPConfig:
    cp_size: int = 1                     # CP 并行度
    cp_engine: str = "vllm_ascend"       # 哪个引擎的 CP 实现
    attn_type: str = "flash_attn"        # 哪种 attention 类型
    cp_kv_cache_interleave_size: int = 128  # KV cache 交错大小
    enable_pcp: bool = True              # 是否启用 PCP
    enable_dcp: bool = True              # 是否启用 DCP
    engine_params: dict = {}             # 引擎特定参数
```

**引擎抽象**：`CPEngine` 是抽象基类，每种引擎实现自己的通信模式：
- `VllmAscendCPEngine`：all_gather + NPU 融合
- `SglangCPEngine`：ring-based + softmax 重算
- `CustomCPEngine`：用户定义

**为什么需要抽象？** 因为不同推理引擎的 CP 实现差异很大。vllm-ascend 用 all_gather（数据全收集），sglang 用 ring（数据环形传递）。如果将来要仿真 sglang 的性能，不能硬编码 vllm-ascend 的逻辑。

### Unit 4：集成到 serving_cast 流水线（RFC §3 Unit 4）

**做什么**：把 CP 仿真真正接入推理流水线，让每次推理仿真时自动计算 CP 开销。

**具体操作**：

1. **扩展 `serving_cast/config.py::ParallelConfig`**：新增 `cp_size` 字段（**注意**：serving_cast 的 ParallelConfig 和 tensor_cast 的是两个不同的类！）

2. **修改 `serving_cast/model_runner.py`**：在初始化 tensor_cast 的 model runner 时，传入 `cp_size` 和 `cp_config` 参数

3. **修改 `tensor_cast/core/model_runner.py`**：在每个 transformer layer 的 attention 子图中，调用 `simulate_cp_overhead()` 加入 CP 通信开销

4. **桥接通信接口**：把 `InterconnectTopology` 的带宽参数转换成 `CommunicationManager.device2device_sync()` 能用的格式

5. **约束检查**：
   - `tp_size × dp_size × pp_size × cp_size == world_size`
   - `ulysses_size > 1` 时跳过 CP 仿真

### Unit 5：端到端验证与 Benchmark（RFC §3 Unit 5）

**做什么**：写测试代码，验证 CP 仿真的正确性。

**要验证的场景**：

| 场景 | CP size | attention 类型 | 序列长度 | 期望结果 |
|------|---------|---------------|----------|----------|
| 纯 Prefill | 2/4/8 | flash_attn/sfa/mla | 32K/128K | 通信开销占总时间 10-30% |
| 纯 Decode | 2/4/8 | flash_attn | 128K context | DCP 通信 << PCP 通信 |
| 混合 Prefill+Decode | 2/4 | flash_attn | 64K | batch 混合计算正确 |
| 引擎切换 | 2 | flash_attn | 32K | vllm_ascend vs sglang 结果不同 |
| Ulysses 冲突 | 2 + ulysses=2 | flash_attn | 32K | 报错拒绝 |

**RFC 原文**：

```
Unit 1 (Config+Group) ──┬──► Unit 3 (Config System) ──► Unit 4 (Integration) ──► Unit 5 (Validation)
                          │
Unit 2 (Comm Modeling) ──┘
```

## 第五章：配置示例（RFC §5）

### 5.1 vllm-ascend 风格标准配置

```yaml
parallel_config:
  world_size: 8     # 总共 8 张 GPU
  tp_size: 2        # 每层切 2 份（Tensor 并行）
  dp_size: 2        # 存 2 份模型副本（Data 并行）
  cp_size: 2        # 序列切 2 份（Context 并行） ← 新增
  pp_size: 1        # 流水线只有 1 段
  ep_size: 1        # 不用专家并行

cp_config:
  cp_engine: "vllm_ascend"    # 用 vllm-ascend 的 CP 实现
  attn_type: "flash_attn"     # 标准 Flash Attention
  enable_pcp: true            # PCP 开
  enable_dcp: true            # DCP 开
```

**验证**：2 × 2 × 1 × 2 = 8，刚好分完。

### 5.2 sglang ring-attention 配置

```yaml
cp_config:
  cp_engine: "sglang"       # 切换到 sglang 风格
  enable_dcp: false         # sglang 通常不支持 DCP
  engine_params:
    ring_degree: 4          # ring attention 的环大小
    chunk_size: 8192        # 每次传递的 KV chunk 大小
```

### 5.3 MLA 模型配置（DeepSeek-V3）

```yaml
cp_config:
  cp_engine: "vllm_ascend"
  attn_type: "mla"          # Multi-head Latent Attention
  engine_params:
    kv_lora_rank: 512       # MLA 的低秩压缩维度
    qk_rope_head_dim: 64    # RoPE 的 head 维度
    v_head_dim: 128         # V 的 head 维度
```

### 5.4 仅 PCP 不用 DCP

```yaml
cp_config:
  enable_pcp: true
  enable_dcp: false    # 短 decode、长 prefill 场景用这个
```

**什么场景用这个？** 如果应用场景是：用户发来很长的问题，但 AI 回复很短（比如翻译、摘要），那么 prefill 阶段需要 CP，但 decode 阶段不需要。

## 第六章：复杂度与风险总结（RFC §6）

| 等级 | Work Unit | 关键风险（白话） |
|------|-----------|----------------|
| Tier 1 | Unit 1 | 改了 rank 网格可能影响已有的并行组合，需要把之前所有并行方式的测例都跑一遍 |
| Tier 2 | Unit 2 | all_gather 开销估计依赖带宽模型精度；MLA 仿真复杂，因为多了一层低秩投影 |
| Tier 2 | Unit 3 | 多引擎抽象增加维护成本；不同引擎的 LSE merge 策略差异很大，需要精确建模 |
| Tier 3 | Unit 4 | **最复杂**：CP + TP + DP + chunked prefill 排列组合太多，边界条件容易遗漏 |
| Tier 2 | Unit 5 | 没有真实硬件 CP 数据做校准（目前只能用解析公式估算） |

## 第七章：开放问题（RFC §7）

1. **PD 分离**：vllm-ascend 支持 Prefill 和 Decode 用不同机器（PD 分离部署）。CP 在 PD 分离场景下，KV cache 需要跨 prefill 节点和 decode 节点传输。当前 RFC **不覆盖**这个场景，留待后续。

2. **CP + EP 组合**：vllm-ascend 的 rank 网格没有独立 EP 维度（EP 通过 transpose 派生）。ascend_msmodeling 是否需要同时包含 CP 和 EP 维度？取决于是否计划同时使用 MoE 模型 + 长序列。

3. **真实数据校准**：仿真精度依赖带宽参数和计算模型，需要后续用 vllm-ascend 真实 profiling 数据来校准。

## 附录：常见问题速查

**Q: PCP 和 DCP 必须一起开吗？**
A: 不必须。可以只开 PCP（长 prefill 场景）、只开 DCP（长 decode 场景）、或同时开。

**Q: Standard PCP 和 Hybrid PCP 选哪个？**
A: Standard（默认）通信量更小。Hybrid 在某些情况下数值精度更好。仿真框架支持通过 `pcp_use_hybrid_attn` 切换。

**Q: CP 和 Ulysses 能同时用吗？**
A: 不能。它们是两种互斥的长序列并行策略。同时启用会报错。

**Q: SFA 和 MLA 的 CP 为什么不一样？**
A: 因为它们的 attention 实现不同。SFA 用稀疏 attention、MLA 用低秩压缩，都需要额外的数据收集步骤。

**Q: `_npu_attention_update` 和 `_npu_attn_out_lse_update` 有什么区别？**
A: 名字很像但完全不同。前者是 decode 专用（common_cp.py:129），后者是 Standard PCP prefill 专用（common_cp.py:158）。两者都调用 `torch_npu.npu_attention_update()`，但输入格式不同。
